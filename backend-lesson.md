# 백엔드 기초 수업 노트

> 컴퓨터 마을 견학 게임과 함께 학습한 백엔드 개념 정리

---

## 1. 요청-응답 흐름 (Request-Response)

```
사용자 (브라우저)  →  서버  →  데이터베이스
       ↑                      ↓
       ←  ←  ←  ←  ←  ←  ←
```

1. 사용자가 브라우저에서 URL을 입력하거나 버튼을 클릭합니다
2. 브라우저가 서버에 **요청(Request)**을 보냅니다
3. 서버가 요청을 처리하고, 필요하면 데이터베이스에서 데이터를 가져옵니다
4. 서버가 **응답(Response)**을 보냅니다
5. 브라우저가 응답을 화면에 표시합니다

**비유:** 택배 주문과 같습니다
- 주문(요청) → 물류센터(서버) → 창고(데이터베이스) → 배송(응답)

---

## 2. API (Application Programming Interface)

API는 **프로그램끼리 대화하는 규약**입니다.

- 백엔드 서버가 제공하는 **데이터 출입문**
- 프론트엔드가 서버에 데이터를 요청하거나 보낼 때 사용하는 **약속된 방법**

**비유:** 레스토랑의 메뉴판
- 메뉴판(API)을 보고 주문하면, 주방(서버)이 요리해서 가져다줍니다
- 주방 안에 들어가서 직접 요리할 수는 없습니다

### REST API
가장 흔한 API 방식. URL로 어떤 자원을 다룰지 정합니다.

| HTTP 메서드 | 의미 | 예시 |
|------------|------|------|
| GET | 데이터 가져오기 | 게시글 목록 조회 |
| POST | 데이터 새로 만들기 | 회원가입 |
| PUT | 데이터 수정하기 | 프로필 수정 |
| DELETE | 데이터 삭제하기 | 게시글 삭제 |

---

## 3. CRUD

데이터를 조작하는 4가지 기본 동작입니다.

| 약어 | 영어 | 의미 | 예시 |
|------|------|------|------|
| C | Create | 만들기 | 새 글쓰기, 회원가입 |
| R | Read | 읽기 | 게시글 보기, 목록 조회 |
| U | Update | 수정하기 | 프로필 수정, 글 수정 |
| D | Delete | 삭제하기 | 글 삭제, 회원 탈퇴 |

모든 웹앱은 이 네 가지로 데이터를 다룹니다.

---

## 4. JSON (JavaScript Object Notation)

**데이터를 주고받는 포맷(형식)**입니다.

```json
{
  "name": "홍길동",
  "age": 25,
  "hobby": ["독서", "코딩"]
}
```

- 키-값 쌍으로 데이터를 표현합니다
- 사람도 읽기 쉽고, 컴퓨터도 처리하기 쉽습니다
- 프론트엔드와 백엔드가 대화할 때 사용하는 **공통 언어**

**비유:** 메모장에 적은 메모
- "이름: 홍길동, 나이: 25, 취미: 독서, 코딩"
- 누구나 쉽게 읽을 수 있는 정리된 형식

## 5. 보안

### 5-1. 인증(Authentication) vs 인가(Authorization)

| 구분 | 의미 | 예시 |
|------|------|------|
| 인증 | **너 누구야?** (신원 확인) | 로그인 (이메일+비밀번호 입력) |
| 인가 | **너 뭘 할 수 있어?** (권한 확인) | 관리자만 게시글 삭제 가능 |

- 인증 먼저 → 인가 그 다음
- 로그인(인증)을 해야, 내 정보 수정(인가)을 할 수 있습니다

### 5-2. 해싱 + 솔팅

비밀번호를 안전하게 저장하는 방법입니다.

- **해싱**: 비밀번호를 긴 문자열로 변환 (예: `1234` → `a1b2c3d4...`)
  - 원래 비밀번호로 되돌릴 수 없습니다
- **솔팅**: 해싱하기 전에 랜덤 문자열을 섞는 것
  - 같은 비밀번호라도 다른 결과가 나와서 더 안전합니다

**왜 필요한가?**
- 데이터베이스가 해킹되어도 비밀번호가 그대로 보이지 않습니다
- `1234`가 아니라 `a1b2c3d4...`만 노출됩니다

### 5-3. 쿠키 vs 세션

로그인 상태를 유지하는 두 가지 방법입니다.

| 구분 | 쿠키 | 세션 |
|------|------|------|
| 저장 위치 | 브라우저 (사용자 컴퓨터) | 서버 |
| 보안 | 낮음 (누구나 볼 수 있음) | 높음 (서버에 저장) |
| 수명 | 설정한 시간 동안 유지 | 브라우저 닫으면 보통 사라짐 |
| 용도 | 자동 로그인, 설정 기억 | 로그인 상태 유지 |

**비유:**
- 쿠키 = 손님에게 주는 **도장 카드** (손님 소지)
- 세션 = 가게 안의 **손님 목록** (가게가 관리)

## 6. 배포

만든 앱을 인터넷에 올려서 누구나 접근할 수 있게 하는 것입니다.

### 프론트엔드 배포

| 서비스 | 특징 |
|--------|------|
| Vercel | Next.js와 궁합 최고, 손이 가장 덜 감 |
| Netlify | Vercel 대안, 상업적 이용 허용 |
| Cloudflare Pages | 대역폭 무제한, 가장 빠름 |
| GitHub Pages | 정적 사이트 전용, 가장 단순 |

### 백엔드 배포

| 서비스 | 특징 |
|--------|------|
| Render | 무료 백엔드 API 서버, 15분 안 쓰면 잠듦 |
| Supabase | DB + 로그인 + API 한 번에 제공 |
| Oracle Always Free | 진짜 리눅스 서버(VM), 카드 필요 |

## 7. 무료 호스팅 옵션 (2026년 기준)

> ⚠️ 정책은 바뀔 수 있으니, 가입 전에 공식 요금 페이지를 확인하세요.

### 뭘 올리느냐에 따라 고르면 됩니다

**HTML/CSS로 만든 화면만 있는 사이트** (포트폴리오, 소개 페이지)
→ GitHub Pages 또는 Cloudflare Pages (가장 간단, 카드 불필요)

**React/Vue 등으로 만든 인터랙티브한 화면**
→ Vercel 또는 Netlify (코드 저장소에서 바로 배포, 카드 불필요)

**로그인·데이터 저장이 필요한 서비스** (서버가 데이터를 처리해야 할 때)
→ Supabase (DB+로그인 한 번에 제공, 카드 불필요)
→ Render (API 서버가 따로 필요할 때, 카드 불필요)

**"진짜 서버를 만져 보며 공부"하고 싶을 때**
→ Oracle Always Free (진짜 컴퓨터를 줌. 단 설정 어려움, 카드 필요)

### 시작 전에 알아 둘 세 가지 함정

1. **잠듦(spin-down)** - 무료 백엔드는 한동안 아무도 안 오면 잠듭니다. Render가 대표적. 다시 깨는 데 ~1분 걸림 (고장 아님)
2. **기한·일시정지** - DB를 공짜로 주는 곳은 30일~90일 안 쓰면 만료되는 곳이 있습니다. Supabase는 7일 미사용 시 일시정지 (데이터는 보존). 중요한 데이터는 백업을 꼭!
3. **요금 폭탄 방지** - 카드를 등록한 곳은 무료 한도를 넘으면 돈이 나갑니다. 미리 예산 상한을 걸어 두면 안전합니다

**정리** - 무료는 "배우고 시연하기"엔 충분합니다. "많은 사람에게 안정적으로 24시간 제공"하는 단계가 오면, 그때 유료를 검토하면 됩니다.

## 핵심 용어 정리

| 용어 | 의미 |
|------|------|
| 프론트엔드 | 사용자가 눈으로 보고 만지는 화면 부분 |
| 백엔드 | 서버에서 데이터를 처리하는 보이지 않는 부분 |
| API | 프로그램끼리 대화하는 규약 |
| REST | URL과 HTTP 메서드로 자원을 다루는 API 방식 |
| CRUD | 데이터 조작 4가지 (만들기, 읽기, 수정, 삭제) |
| JSON | 데이터를 주고받는 포맷 |
| 인증 | 신원 확인 (너 누구야?) |
| 인가 | 권한 확인 (너 뭘 할 수 있어?) |
| 해싱 | 비밀번호를 문자열로 변환 |
| 솔팅 | 해싱 전 랜덤 문자열을 섞는 것 |
| 쿠키 | 브라우저에 저장되는 작은 데이터 |
| 세션 | 서버에 저장되는 로그인 상태 |
| 배포 | 앱을 인터넷에 올리는 것 |
