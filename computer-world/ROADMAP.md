# 컴퓨터 월드 교육 로드맵

> **프로젝트 목적**: 컴퓨터를 전혀 모르는 사람이 "컴퓨터가 실제로 어떻게 작동하는가"를 이해할 수 있는 인터랙티브 교과서

---

## 전체 학습 여정 (World 순서)

```
실세계 (현재 위치)
    ↓
하드웨어 월드 (구현 완료)
    ↓
운영체제 월드
    ↓
브라우저 / 프론트엔드 월드
    ↓
자바스크립트 월드
    ↓
백엔드 / 서버 월드
    ↓
데이터베이스 월드
    ↓
네트워크 월드 (구현 완료)
    ↓
보안 월드
    ↓
배포 월드
    ↓
API 세계 (연결 고리)
```

## World 0: 실세계 (Real World)

**현재 상태**: 구현 완료

**교육적 목적**: 사용자가 현재 있는 곳에서 시작하여, 컴퓨터 본체를 클릭하면 하드웨어 월드로 진입

**주요 개념**:
- 컴퓨터 본체 = 하드웨어의 집합체
- 모니터, 키보드, 마우스 = 입출력 장치

## World 1: 하드웨어 월드 (Hardware World)

**현재 상태**: 구현 완료

**교육적 목적**: 컴퓨터의 물리적 구조 이해 - 어떤 부품이 있고, 서로 어떻게 연결되는가

### 구현된 건물과 개념

| 건물 | 기술 용어 | 핵심 설명 |
|---|---|---|
| 메인보드 (기판) | Mainboard, PCB | 모든 부품을 연결하는 회로 기판 |
| CPU (중앙처리장치) | Central Processing Unit | 계산과 처리를 담당하는 핵심 칩 |
| CPU 내장 캐시 | Cache (L1/L2/L3) | CPU 안의 가장 빠른 기억 (SRAM) |
| RAM (메모리) | Random Access Memory | 임시로 데이터를 보관하는 장치 (DRAM) |
| 저장장치 (SSD/HDD) | Storage | 데이터를 영구 보관하는 장치 |
| GPU (그래픽처리장치) | Graphics Processing Unit | 화면을 그리는 전문 칩 |
| 운영체제 (OS) | Operating System | 하드웨어를 관리하는 소프트웨어 |

### 핵심 관계
- CPU → 캐시 → RAM → 저장장치 (속도와 용량의 계층)
- CPU는 메인보드의 소켓에 꽂힘
- RAM은 DIMM 슬롯에 수직으로 꽂힘
- GPU는 PCIe 슬롯에 확장 카드로 꽂힘
- 저장장치는 케이블로 연결 (보드 밖)
- OS는 메모리에 상주하며 하드웨어를 관리

### 아직 추가해야 할 개념 (OS世界에서 심화)
- 프로세스 vs 스레드
- 시분할 (Time-sharing)
- 동기/비동기 처리
- 멀티태스킹

## World 2: 운영체제 월드 (OS World) — 미구현

**교육적 목적**: 소프트웨어가 하드웨어를 어떻게 통제하는가 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 조절소 (커널) | Kernel | OS의 핵심 - 하드웨어와 프로그램 사이의 중재자 |
| 관제 센터 (스케줄러) | Scheduler | CPU의 시간을 프로세스에게 골고루 배분 |
| 메모리 창고 (메모리 관리) | Memory Manager | RAM 사용량을 관리 |
| 서류 보관소 (파일 시스템) | File System | 저장장치의 파일을 정리하고 관리 |
| 출입국 관리 (I/O 관리) | I/O Manager | 키보드, 마우스 등 장치의 입출력 처리 |

### 핵심 개념

**프로세스 vs 스레드**
- 프로세스 = 실행 중인 프로그램 (은행 지점 하나)
- 스레드 = 프로세스 안의 실제 작업 단위 (지점 안 창구 직원)
- CPU는 프로세스가 아니라 스레드를 실행

**시분할 (Time-sharing)**
- 코어가 여러 개인데도 스레드는 수백 개
- 운영체제가 아주 짧은 시간씩 번갈아 실행
- "한 번에 한 명만 서랍 열기" 같은 규칙 필요

**동기 vs 비동기**
- 동기 = 전화 통화 (상대가 받을 때까지 기다림)
- 비동기 = 문자 (보내놓고 하던 일 계속)
- 자바스크립트는 비동기로 처리하여 화면 멈춤 방지

**멀티프로세싱 vs 멀티스레딩**
- 멀티프로세싱 = 완전히 분리된 은행 지점 여러 개
- 멀티스레딩 = 한 지점 안 여러 창구 직원

## World 3: 브라우저 / 프론트엔드 월드 (Browser/Frontend World) — 미구현

**교육적 목적**: 웹페이지가 3계층으로 이루어져 있음을 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 골조 공장 (HTML) | HyperText Markup Language | 웹페이지의 구조와 의미를 정의 |
| 인테리어 공방 (CSS) | Cascading Style Sheets | 웹페이지의 표현과 꾸밈을 담당 |
| 동력실 (JavaScript) | JavaScript | 웹페이지의 동작과 상호작용을 담당 |

### 핵심 개념

**HTML (구조)**
- 1989년 팀 버너스리가 문서 연결을 위해 만듦
- `<h1>`은 "이건 가장 큰 제목", `<button>`은 "이건 누르는 버튼"
- 검토 기준: 필요한 정보와 업무 용어가 빠지지 않았는가

**CSS (표현)**
- 내용과 꾸밈을 분리하여 수정 용이성 확보
- "캐스케이딩" = 여러 스타일이 겹칠 때 우선순위가 폭포처럼 층을 이룸
- 검토 기준: 눈이 어디로 가는지, 깨지는 구간은 없는가

**JavaScript (동작)**
- 1995년 브렌던 아이크가 열흘 만에 만듦
- 첫 임무: 폼 검증 (입력값 확인)
- 검토 기준: 클릭·입력·상태 변화가 기대대로 일어나는가

**관심사의 분리 (Separation of Concerns)**
- 복잡한 문제를 다룰 때 한 번에 한 측면에만 집중
- 웹의 3계층은 이 원칙이 화면 설계에 적용된 결과

## World 4: 자바스크립트 월드 (JavaScript World) — 미구현

**교육적 목적**: 자바스크립트의 동작 원리와 비동기 처리 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 일꾼 (메인 스레드) | Main Thread | 자바스크립트가 실제로 실행되는 곳 |
| 대기줄 (콜백 큐) | Callback Queue | 끝난 비동기 작업이 기다리는 곳 |
| 순환 감독 (이벤트 루프) | Event Loop | 대기줄을 계속 돌면서 결과를 처리 |
| 뒷방 (브라우저 API) | Web APIs | fetch, setTimeout 등 오래 걸리는 일을 처리 |

### 핵심 개념

**싱글 스레드**
- 자바스크립트는 기본적으로 창구(스레드) 하나만 사용
- 그래서 동기 방식으로만 하면 화면이 멈춤

**이벤트 루프 (Event Loop)**
- 오래 걸리는 일은 브라우저에게 맡기고 다음 일로 넘어감
- 브라우저가 처리를 끝내면 결과를 대기줄에 올려둠
- 일꾼이 짬이 날 때 대기줄을 확인하여 결과를 처리

**비동기 패턴**
- `async/await`: 기다리는 코드를 읽기 쉽게 만듦
- `Promise`:未来的 결과를 담는 컨테이너
- `setTimeout`: 일정 시간 후 실행

## World 5: 백엔드 / 서버 월드 (Backend World) — 미구현

**교육적 목적**: 눈에 보이지 않는 서버 쪽에서 무슨 일이 일어나는가 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 서버 빌딩 | Server | 데이터를 처리하고 저장하는 컴퓨터 |
| 요청 창구 | Request Handler | 클라이언트의 요청을 받아 처리 |
| 응답 배달부 | Response Sender | 처리 결과를 클라이언트에게 돌려줌 |
| 인증 경비원 | Authentication | 로그인한 사용자인지 확인 |
| API 창구 | API Endpoint | 정해진 형식으로 데이터를 주고받는 창구 |

### 핵심 개념

**클라이언트-서버 모델**
- 클라이언트(브라우저)가 요청 → 서버가 응답
- HTTP 프로토콜로 통신

**REST API**
- 리소스(데이터)를 URL로 표현
- GET(읽기), POST(쓰기), PUT(수정), DELETE(삭제)

**인증 (Authentication)**
- 비밀번호, 토큰(JWT), OAuth 등으로 사용자 확인
- 세션 vs 토큰 기반 인증

## World 6: 데이터베이스 월드 (Database World) — 미구현

**교육적 목적**: 데이터가 어떻게 저장되고 관리되는가 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 서류 창고 (DBMS) | Database Management System | 데이터를 체계적으로 저장/관리 |
| 검색 도서관 (인덱스) | Index | 데이터를 빠르게 찾기 위한 색인 |
| 백업 창고 | Backup | 데이터가 날아가지 않도록 복사본 보관 |

### 핵심 개념

**관계형 데이터베이스 (RDBMS)**
- 테이블(표) 형태로 데이터 저장
- SQL로 데이터를 조작
- MySQL, PostgreSQL, SQLite 등

**NoSQL**
- 테이블 외 다양한 형태로 저장
- MongoDB, Redis 등

**정규화**
- 데이터 중복을 줄이고 일관성 유지

## World 7: 네트워크 월드 (Network World)

**현재 상태**: 구현 완료

**교육적 목적**: 데이터가 인터넷을 통해 어떻게 이동하는가 이해

### 구현된 건물과 개념

| 건물 | 기술 용어 | 핵심 설명 |
|---|---|---|
| DNS (이름서버) | Domain Name System | 이름을 숫자 주소로 변환 |
| 라우터 (공유기) | Router | 데이터 경로를 결정 |
| 방화벽 (화재벽) | Firewall | 보안 위협을 차단 |
| HTTP (규약) | HyperText Transfer Protocol | 웹 데이터를 전송하는 통신 규칙 |

### 핵심 관계
- DNS는 라우터를 통해 질문
- 라우터는 방화벽과 함께 작동
- 방화벽이 허용해야 HTTP 데이터가 오갈 수 있음

## World 8: 보안 월드 (Security World) — 미구현

**교육적 목적**: 컴퓨터 보안의 중요性和 기본 원리 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 암호화 공장 | Encryption | 데이터를 암호화하여 안전하게 보관 |
| 인증서 관리소 | Certificate | HTTPS, SSL/TLS 인증서 |
| 보안 감시실 | Security Monitor | 위협을 감지하고 대응 |

### 핵심 개념

**암호화 (Encryption)**
- 대칭키: 하나의 키로 암복호화
- 비대칭키: 공개키와 개인키 쌍
- 해시: 단방향 변환 (비밀번호 저장 등)

**HTTPS**
- HTTP + TLS/SSL 암호화
- 웹사이트 인증서로 신원 확인

**보안 위협**
- 피싱(Phishing), 멀웨어(Malware), 랜섬웨어
- SQL 인젝션, XSS, CSRF

## World 9: 배포 월드 (Deployment World) — 미구현

**교육적 목적**: 만든 웹사이트를 실제로 인터넷에 올리는 과정 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 호스팅 센터 | Hosting | 웹사이트를 저장하고-serving하는 곳 |
| 도메인 관리소 | Domain | 웹사이트 주소를 관리 |

### 핵심 개념

**호스팅**
- 공유호스팅, VPS, 클라우드(AWS, Azure, GCP)

**도메인**
- 도메인 등록 (1년에 약 만원)
- DNS 설정

**GitHub Pages**
- Git 저장소에서 자동으로 웹사이트 배포
- 무료이고 간편함

## World 10: API 세계 (API World) — 미구현

**교육적 목적**: 프로그램끼리 어떻게 대화하는가 이해

### 구현해야 할 건물과 개념

| 건물 (가칭) | 기술 용어 | 핵심 설명 |
|---|---|---|
| 통역사 (API) | Application Programming Interface | 프로그램끼리 대화하는 규칙 |
| 요청 서랍 | Request | 데이터를 요청하는 형식 |
| 응답 서랍 | Response | 요청에 대한 답변 |

### 핵심 개념

**API란**
- 서로 다른 프로그램끼리 데이터를 주고받는 통신 방식
- 레스토랑의 웨이터처럼 중간에서 역할
- API ≠ AI (발음 비슷하지만 완전히 다른 개념)

**REST API**
- 리소스를 URL로 표현
- HTTP 메서드(GET, POST, PUT, DELETE) 사용

**API 문서**
- API 사용 방법을 정리한 문서
- "이렇게 물어봐야 한다"는 고유한 질문 양식

## 학습 순서 설계 원리

### 1. 물리적 구조 → 논리적 구조 → 동작 원리
- 하드웨어(물리) → OS(논리) → 프로그래밍(동작)

### 2. 눈에 보이는 것 → 보이지 않는 것
- 프론트엔드(눈에 보임) → 백엔드(보이지 않음) → 데이터베이스(보이지 않음)

### 3. 단일 기능 → 연결된 시스템
- 각 세계 독립 학습 → API로 연결되는 전체 시스템 이해

### 4. 사용자 경험 → 내부 원리
- 브라우저에서 보이는 것 → 내부에서 무슨 일이 일어나는가

## 데이터 흐름 데모 (전체 시스템 연결)

### 예시: 웹사이트 접속 과정

```
1. 사용자가 브라우저에 주소 입력
2. DNS가 이름을 숫자 주소로 변환
3. 라우터가 경로 결정
4. 방화벽이 요청 검증
5. HTTP로 서버에 요청 전달
6. 서버가 요청 처리
7. 데이터베이스에서 데이터 조회
8. 서버가 응답 생성
9. 브라우저가 HTML/CSS/JS 수신
10. 브라우저가 화면에 렌더링
```

### 예시: 로그인 과정

```

4. 데이터베이스에서 사용자 정보 조회
5. 서버가 토큰 발급
6. 브라우저가 토큰 저장
7. 이후 요청마다 토큰 포함
```
